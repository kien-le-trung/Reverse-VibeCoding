# Templates

Starter code templates live here.

Templates are designed around two ideas:

1. A future `rev-vib init` command will select a project combination and copy it into `sandbox/`.
2. Templates should be reusable educational building blocks, not one-off generated projects.

The current supported stacks are:

- FastAPI
- React Native

Full-stack projects do not need a separate template family. A full-stack project is generated by composing one backend template and one frontend template, then applying wiring overlays.

## Template Model

Generation uses ordered tree composition. The generator copies directories in order, and later trees may overwrite earlier files.

```text
Stack Base
+ Completeness Level Overlay
+ Domain Overlay
+ Scenario Overlay
+ Bug Seed Overlay
+ Wiring Overlay
+ Project Metadata
```

Example:

```text
FastAPI level_2
+ todo_app
+ pagination
+ off_by_one_bug
```

The copy model is intentionally simple:

```text
copy_tree(base/files, sandbox/project)
copy_tree(level/files, sandbox/project)
copy_tree(domain/files, sandbox/project)
copy_tree(database/files, sandbox/project)
copy_tree(scenario/files, sandbox/project)
copy_tree(wiring/files, sandbox/project)
```

Each layer owns a coherent tree of files. If `level_2` provides `backend/app/main.py`, it replaces the `backend/app/main.py` from `base`. This is expected and preferred over line-level patching.

The generator can then copy the resolved files into:

```text
templates/
  backend/
    fastapi/
  frontend/
    react_native/

sandbox/
  generated_project/
```

## Completeness Levels

Completeness levels are cumulative, stack-specific overlays on top of a shared stack base.

That matters because APIs, schemas, database models, navigation, forms, and state management are core learning objectives. Each level should expose a different amount of professional structure.

Levels should be defined by student responsibility, not only by which files exist.

```text
Level 1 - Student designs most structure.
Level 2 - Student implements behavior inside provided API or UI boundaries.
Level 3 - Student wires persistence, state, or external contracts into existing architecture.
Level 4 - Student maintains, debugs, and refactors an existing app.
```

Each stack maps those responsibilities into concrete files and checks. A level does not need to duplicate the complete base project, but after applying all required lower layers, the result must be a coherent runnable project tree.

## Stack Base

Every stack should have a `base/` directory containing files that every project for that stack receives.

Examples:

- dependency configuration
- app entry point
- minimal health or render baseline
- baseline tests or checks
- shared README fragments

Levels should add or replace files from `base/`. They should not duplicate entire projects unless there is no cleaner option.

### FastAPI Levels

#### Level 1 - Minimal

The student receives almost nothing beyond a runnable project shell.

Includes:

- `pyproject.toml`
- `app/main.py`
- `/health`
- one smoke test

Learns:

- app startup
- routing basics
- test execution
- project layout

#### Level 2 - API

The student receives API routing structure but little business logic.

Includes:

- route modules
- request/response schemas
- placeholder handlers
- failing tests for endpoint behavior

Learns:

- REST design
- validation boundaries
- status codes
- API-level tests

#### Level 3 - Persistence

The student receives database structure and must connect behavior correctly.

Includes:

- database session setup
- initial models
- repository placeholders
- migration or schema setup notes

Learns:

- persistence boundaries
- constraints
- data modeling
- integration testing

#### Level 4 - Service Architecture

The student receives a more realistic codebase with existing decisions.

Includes:

- routes
- schemas
- services
- repositories
- seeded bugs or incomplete features

Learns:

- maintenance work
- refactoring
- debugging
- code review

## Database Learning Options

FastAPI database overlays should be explicit learning options.

Initial options:

- SQLite
- local PostgreSQL
- Supabase Postgres

This is sound because each option teaches a different level of operational maturity.

### SQLite

SQLite should be the default early-path database.

Best for:

- Level 1 and Level 2 projects
- local-first learning
- low setup overhead
- testing persistence concepts without infrastructure friction

Teaches:

- schema design
- constraints
- basic persistence
- repository boundaries
- test database isolation

### Local PostgreSQL

Local PostgreSQL should be introduced when the student is ready for production-like relational behavior.

Best for:

- Level 3 and Level 4 projects
- realistic integration testing
- SQL behavior closer to production

Teaches:

- connection configuration
- migrations
- stronger relational constraints
- database setup documentation
- environment-specific configuration

### Supabase Postgres

Supabase Postgres should be treated as a hosted Postgres learning option, not a completely different database category.

Best for:

- later Level 3 and Level 4 projects
- cloud configuration practice
- projects involving auth or hosted persistence

Teaches:

- managed database setup
- environment variables and secrets
- remote connection constraints
- hosted service tradeoffs

Supabase should not be required for core MVP generation because it introduces account setup and network dependency. It is a good optional learning path once local project generation is stable.

### Later Database Options

MongoDB and DuckDB are reasonable future additions, but they should be introduced as framework-practice variants rather than initial MVP options.

- MongoDB teaches document modeling, denormalization, and different consistency tradeoffs.
- DuckDB teaches analytical workflows, local data processing, and query-heavy data applications.

They should come after the relational path is stable because they change the learning surface substantially.

### React Native Levels

#### Level 1 - Minimal

The student receives a runnable Expo shell.

Includes:

- app entry point
- basic screen
- minimal styling
- smoke test or static check

Learns:

- project startup
- screen rendering
- component basics

#### Level 2 - Navigation

The student receives screen organization and navigation structure.

Includes:

- app layout
- multiple screens
- placeholder domain screens

Learns:

- navigation flow
- screen responsibilities
- UI composition

#### Level 3 - API Client

The student receives a frontend that can communicate with a backend.

Includes:

- API client module
- loading state
- error state
- environment config placeholder

Learns:

- network boundaries
- async state
- frontend/backend contracts

#### Level 4 - State And Forms

The student receives a more realistic mobile app surface.

Includes:

- forms
- domain state
- validation hooks
- seeded UI or state bugs

Learns:

- form modeling
- local state
- accessibility
- maintainable feature structure

## Full-Stack Composition

Full-stack generation should compose stack templates instead of introducing a third template type.

Example output:

```text
sandbox/my_project/
  backend/
    ...
  mobile/
    ...
```

The generator should apply a wiring overlay that adds:

- backend URL config
- mobile API client defaults
- shared README instructions
- contract expectations between frontend and backend

This keeps FastAPI and React Native reusable independently while still supporting full-stack projects.

The wiring overlay should live under:

```text
templates/
  wiring/
    react_native_fastapi/
```

This gives full-stack composition a concrete home without creating a separate full-stack template family.

## Overlay Manifests

Each level, domain, scenario, database option, bug seed, and wiring overlay should include an `overlay.yaml` manifest.

Example:

```yaml
id: pagination
type: scenario
stacks:
  - fastapi
requires:
  min_level: level_2
conflicts: []
student_visible: true
```

The manifest should let the generator reject invalid combinations before copying files.

Useful fields:

- `id`
- `type`
- `stacks`
- `requires`
- `conflicts`
- `student_visible`
- `hidden_from_student`
- `files`
- `checks`
- `composition`
- `overwrites`
- `python_dependencies`
- `npm_dependencies`
- `npm_dev_dependencies`
- `maven_dependencies`

Stack base manifests should declare their package dependencies. `rev-vib init` uses these fields to create `requirements.txt`, initialize `.venv`, and record frontend and backend dependency metadata.

## Overlay Layout

Overlays should contain copyable file trees.

```text
overlays/scenarios/pagination/
  overlay.yaml
  files/
  tests/
```

- `files/` contains files copied directly into the generated project.
- `tests/` contains visible tests when the student should see them.
- hidden tests should live outside student-visible generated files.
- overwrites are allowed when a later layer needs to replace an earlier file.

This keeps overlay behavior explicit and avoids a patch engine.

## Generated Output Contract

The generator should produce a stable shape.

Backend-only example:

```text
sandbox/my_project/
  backend/
    ...
  .rv/
    project.json
    hidden_manifest.json
```

Mobile-only example:

```text
sandbox/my_project/
  mobile/
    ...
  .rv/
    project.json
    hidden_manifest.json
```

Full-stack example:

```text
sandbox/my_project/
  backend/
    ...
  mobile/
    ...
  .rv/
    project.json
    hidden_manifest.json
```

## Design Rules

- Every level must have a verifiable baseline.
- Every generated project must have at least one meaningful test or check.
- Incomplete code should be intentional and tied to a task.
- Seeded bugs should be documented outside student-visible files.
- Domain overlays should avoid duplicating whole stack templates.
- Scenario overlays should be small and composable.
- Database overlays should be selected deliberately because they change setup burden and learning goals.
- Invalid overlay combinations should fail during generation, not after files are copied.
